# Phase 22: Ongoing Growth Roadmap - Research

**Researched:** 2026-03-27
**Domain:** Prisma schema migration, DAL extension, Next.js server actions, React conditional rendering
**Confidence:** HIGH

## Summary

Phase 22 adds a second lifecycle state to the milestone system. Currently all milestones are of one implicit type: setup milestones (the 6-step STANDARD_MILESTONES template). This phase introduces a second type — GROWTH milestones — with auto-generation logic triggered when setup completes, plus admin tooling to manage them and conditional client-side rendering.

The codebase is a Next.js 15 app using Prisma + PostgreSQL, the DAL pattern (`lib/dal.ts` with `cache()`), Zod-validated server actions, and client components that receive pre-serialized data from async server components. The stack is established and stable; no new libraries are needed.

The core technical problem is **discriminating between milestone types** throughout the stack. Currently `getMilestones()` returns all milestones for a client with no type distinction. Phase 22 requires splitting them into SETUP vs GROWTH buckets, auto-generating 12 GROWTH milestones on setup completion, and showing the correct set on the client dashboard.

**Primary recommendation:** Add a `milestoneType` enum field (`SETUP | GROWTH`) to the Prisma `Milestone` model. This is the simplest discriminator with zero ambiguity, backward-compatible with a default value of `SETUP`, and queryable directly in Prisma without application-layer filtering.

### Pattern 1: Prisma Enum for Milestone Type
**What:** Add a `MilestoneType` enum to the schema with a default of `SETUP`.
**When to use:** Whenever you need to discriminate between milestone categories in queries.

```typescript
// Source: prisma/schema.prisma (existing file, extend it)
enum MilestoneType {
  SETUP
  GROWTH
}

model Milestone {
  // ... existing fields ...
  milestoneType  MilestoneType   @default(SETUP)
}
```

This is backward-compatible. All 6 standard milestones created via `seedStandardMilestones()` will receive `SETUP` as their default with no migration data backfill needed.

### Pattern 2: Auto-generation Triggered in updateMilestones()
**What:** After the milestone batch update transaction commits, check if setup is now complete and no GROWTH milestones exist yet. If so, create 12 monthly GROWTH milestones.
**When to use:** Only triggered once — guard with `milestoneType: 'GROWTH'` count check.

```typescript
// Source: app/admin/clients/[clientId]/actions.ts (existing file, extend updateMilestones)

// After prisma.$transaction(updates) succeeds:
const allSetupMilestones = await prisma.milestone.findMany({
  where: { clientId, milestoneType: 'SETUP' },
  select: { status: true },
})
const setupComplete =
  allSetupMilestones.length >= 6 &&
  allSetupMilestones.every((m) => m.status === 'COMPLETED')

if (setupComplete) {
  const existingGrowth = await prisma.milestone.count({
    where: { clientId, milestoneType: 'GROWTH' },
  })
  if (existingGrowth === 0) {
    await autoGenerateGrowthMilestones(clientId)
  }
}
```

### Pattern 3: "First of Next Month" Date Calculation
**What:** Calculate the start date for the first GROWTH milestone as the 1st of the month following today's date. Subsequent months increment by 1 month each.

```typescript
// Pure Date arithmetic — no external library needed
function firstOfNextMonth(from: Date): Date {
  return new Date(from.getFullYear(), from.getMonth() + 1, 1)
}

// Generate 12 monthly review milestones
function buildGrowthMilestones(clientId: string) {
  const base = firstOfNextMonth(new Date())
  return Array.from({ length: 12 }, (_, i) => ({
    clientId,
    title: `Monthly Review — ${new Date(base.getFullYear(), base.getMonth() + i, 1)
      .toLocaleDateString('en-US', { month: 'long', year: 'numeric' })}`,
    description: 'Monthly performance review and optimization session.',
    milestoneType: 'GROWTH' as const,
    order: i + 1,
    status: 'NOT_STARTED' as const,
    progress: 0,
    dueDate: new Date(base.getFullYear(), base.getMonth() + i, 1),
    notes: [],
  }))
}
```

### Pattern 4: setupComplete Derivation (established in Phase 19)
The admin page already has this logic. The exact guard from `app/admin/page.tsx` (line 36-39):
```typescript
const setupMilestones = client.milestones.filter((m) => m.order <= 6)
const setupComplete =
  setupMilestones.length >= 6 &&
  setupMilestones.every((m) => m.status === 'COMPLETED')
```
With the new `milestoneType` field, this should be updated to filter on `milestoneType === 'SETUP'` rather than `order <= 6`. This is more robust (admin could add GROWTH milestones that have order > 6 but the SETUP check should still work correctly).

### Pattern 5: Conditional Roadmap Rendering
**What:** `DashboardOverview` already renders the Growth Roadmap section with `orderedMilestones`. Phase 22 must split the milestones prop or add a `setupComplete` boolean prop so the component can decide which section to show.

Two approaches — **use approach A**:

**A. Pass both milestone arrays and a flag (preferred):** Server component computes `setupComplete` from the SETUP milestones and passes it down. `DashboardOverview` renders the correct section based on `setupComplete`.

**B. Single array with type field:** Pass all milestones and filter client-side. Simpler but means the client component contains business logic about setup completion. Avoid this.

### Anti-Patterns to Avoid
- **Using `order <= 6` to identify setup milestones:** Fragile once GROWTH milestones exist. Use `milestoneType === 'SETUP'` instead.
- **Auto-generating inside a Prisma transaction with the update:** `autoGenerateGrowthMilestones` should run AFTER the main transaction commits to avoid extending transaction duration unnecessarily.
- **Re-generating on every save:** Always count existing GROWTH milestones and skip if `existingGrowth > 0`.
- **Filtering milestones client-side in DashboardOverview:** The server component should pass separate arrays (setupMilestones, growthMilestones) so client component stays presentation-only.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Type discrimination | Custom tag fields (e.g. `isGrowth: Boolean`) | Prisma enum `MilestoneType` | Enums are indexed, type-safe, queryable |
| Date arithmetic for "1st of next month" | External date library | Native `new Date(y, m+1, 1)` | No edge cases for this simple requirement |
| Admin add/remove GROWTH milestone | New UI framework | Extend existing `MilestoneEditTable` pattern | Already has server action wiring, toast feedback, `useTransition` |

**Key insight:** The entire pattern for server actions (Zod validation, `verifySession`, `revalidatePath`) is established. GROWTH milestone actions are new implementations of the same pattern, not new patterns.

## Common Pitfalls

### Pitfall 1: Vacuous Truth on setupComplete
**What goes wrong:** `milestones.every(m => m.status === 'COMPLETED')` returns `true` for an empty array.
**Why it happens:** JavaScript `.every()` on empty array is vacuously true.
**How to avoid:** The Phase 19 guard (`setupMilestones.length >= 6 && ...`) must be preserved. With milestoneType filtering, the guard becomes `setupMilestones.length >= 6 && setupMilestones.every(...)`.
**Warning signs:** A new client with 0 milestones shows the growth roadmap.

### Pitfall 2: Duplicate Growth Milestone Generation
**What goes wrong:** Saving milestones a second time after setup is complete generates 12 more GROWTH milestones.
**Why it happens:** The completion check fires every time all setup milestones are COMPLETED.
**How to avoid:** Guard with `existingGrowth === 0` before creating. This must be an atomic check-then-create to handle race conditions — use a Prisma transaction or simply rely on the single-admin nature of this app (race condition risk is effectively zero).
**Warning signs:** Client has 24, 36, etc. growth milestones.

### Pitfall 3: Milestone Order Collision
**What goes wrong:** GROWTH milestones use `order: 1-12` which collides with SETUP milestones' `order: 1-6`.
**Why it happens:** `order` is used for display sorting within the current context (setup OR growth), not globally unique.
**How to avoid:** Sort by `order` within `milestoneType === 'GROWTH'` queries only. OR use `order: 101-112` for growth milestones. The cleanest solution is ordering within filtered sets. The DAL should always filter by type before applying `orderBy: { order: 'asc' }`.
**Warning signs:** Growth roadmap shows interleaved setup and growth cards in wrong order.

### Pitfall 4: getMilestones() Breaking Client Dashboard
**What goes wrong:** `getMilestones()` in `lib/dal.ts` currently returns all milestones. After the schema change, it will return both SETUP and GROWTH milestones mixed together. The existing `DashboardOverview` roadmap section would show all 18 milestones (6 setup + 12 growth) in one horizontal scroll.
**Why it happens:** The function doesn't filter by `milestoneType`.
**How to avoid:** In `app/dashboard/page.tsx`, fetch milestones then partition client-side, OR update the DAL to expose separate functions. The cleanest is to either add type-specific DAL functions OR have the page partition the array before passing props.
**Warning signs:** Client sees 18 cards in the roadmap instead of the correct 6 or 12.

### Pitfall 5: Serialization of New Enum Field
**What goes wrong:** `milestoneType` is returned from the DB but not included in the serialized milestone objects passed from server to client component.
**Why it happens:** The serialization in `app/dashboard/page.tsx` uses spread `...m` so any new fields are included automatically — BUT the `SerializedMilestone` interface in `dashboard-overview.tsx` must also be updated, and `lib/types/milestone.ts` must add the `MilestoneType` enum.
**Warning signs:** TypeScript errors on `m.milestoneType` in the client component.

## Code Examples

### Add Growth Milestone Server Action Pattern
```typescript
// Source: following existing pattern from app/admin/clients/[clientId]/actions.ts

const addGrowthMilestoneSchema = z.object({
  clientId: z.string().cuid(),
  title: z.string().min(1).max(200).trim(),
  dueDate: z.string().datetime().nullable(),
})

export async function addGrowthMilestone(clientId: string, rawData: unknown) {
  const { userRole } = await verifySession()
  if (userRole !== 'ADMIN') return { error: 'Unauthorized' }

  const validation = addGrowthMilestoneSchema.safeParse({ clientId, ...rawData })
  if (!validation.success) return { error: 'Invalid data' }

  const { title, dueDate } = validation.data

  // Find max order among existing GROWTH milestones for this client
  const maxOrder = await prisma.milestone.aggregate({
    where: { clientId, milestoneType: 'GROWTH' },
    _max: { order: true },
  })

  await prisma.milestone.create({
    data: {
      clientId,
      title,
      milestoneType: 'GROWTH',
      status: 'NOT_STARTED',
      progress: 0,
      order: (maxOrder._max.order ?? 0) + 1,
      dueDate: dueDate ? new Date(dueDate) : null,
      notes: [],
    },
  })

  revalidatePath(`/admin/clients/${clientId}`)
  revalidatePath('/dashboard')
  return { success: true }
}
```

### Remove Growth Milestone Server Action Pattern
```typescript
// Source: following existing deleteNote pattern in same file

const removeGrowthMilestoneSchema = z.object({
  clientId: z.string().cuid(),
  milestoneId: z.string().cuid(),
})

export async function removeGrowthMilestone(clientId: string, milestoneId: string) {
  const { userRole } = await verifySession()
  if (userRole !== 'ADMIN') return { error: 'Unauthorized' }

  const validation = removeGrowthMilestoneSchema.safeParse({ clientId, milestoneId })
  if (!validation.success) return { error: 'Invalid input' }

  // Verify the milestone belongs to this client AND is a GROWTH milestone
  const milestone = await prisma.milestone.findUnique({
    where: { id: milestoneId },
    select: { clientId: true, milestoneType: true },
  })

  if (!milestone) return { error: 'Milestone not found' }
  if (milestone.clientId !== clientId) return { error: 'Milestone does not belong to this client' }
  if (milestone.milestoneType !== 'GROWTH') return { error: 'Cannot remove setup milestones' }

  await prisma.milestone.delete({ where: { id: milestoneId } })

  revalidatePath(`/admin/clients/${clientId}`)
  revalidatePath('/dashboard')
  return { success: true }
}
```

## Open Questions

1. **What happens to existing setup milestone `order` values when the new field is added?**
   - What we know: All existing milestones default to `milestoneType: SETUP` via Prisma default.
   - What's unclear: Whether any clients currently have more or fewer than 6 milestones (data quality).
   - Recommendation: The `setupMilestones.length >= 6` guard handles this — clients with < 6 milestones will never show as setup complete.

2. **Should admin be able to re-order growth milestones?**
   - What we know: The `order` field exists on milestones; no reorder UI exists yet.
   - What's unclear: Not specified in requirements.
   - Recommendation: Out of scope for this phase. Auto-generated milestones use sequential order (1-12). Admin add creates order = max + 1.

3. **Should `getMilestones()` (the original function) be updated or preserved?**
   - What we know: It's used in `app/dashboard/page.tsx` and returns ALL milestones.
   - What's unclear: Whether any other callers need all milestones.
   - Recommendation: Keep `getMilestones()` as-is for backward compatibility. Add `getSetupMilestones()` and `getGrowthMilestones()` as new functions. Update `app/dashboard/page.tsx` to call the new typed functions instead.

4. **How should the admin MilestoneEditTable handle growth milestones?**
   - What we know: Current table shows all milestones; admin uses it to set status/dates/notes.
   - What's unclear: Whether growth milestones should use the same table UI or a separate section.
   - Recommendation: Extend the existing `MilestoneEditTable` with a second section below the setup table labeled "Growth Milestones" with add/remove controls. Keeps all milestone management in one place (the existing `/admin/clients/[clientId]` page).
