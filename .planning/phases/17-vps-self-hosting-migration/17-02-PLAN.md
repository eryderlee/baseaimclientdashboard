  <files>.github/workflows/deploy.yml</files>
  <action>
    Create `.github/workflows/deploy.yml` (create the `.github/workflows/` directory first).

    The workflow must:

    1. **Trigger:** On push to `main` branch only.

    2. **Build job:**
       - `runs-on: ubuntu-latest`
       - Checkout code with `actions/checkout@v4`
       - Setup Node.js 20 with `actions/setup-node@v4` and `cache: 'npm'`
       - `npm ci`
       - `npm run build` with these env vars:
         - `SKIP_ENV_VALIDATION: '1'` (build doesn't have runtime secrets)
         - `SENTRY_AUTH_TOKEN: ${{ secrets.SENTRY_AUTH_TOKEN }}`
         - `SENTRY_ORG: ${{ secrets.SENTRY_ORG }}`
         - `SENTRY_PROJECT: ${{ secrets.SENTRY_PROJECT }}`
         - `NEXT_PUBLIC_APP_URL: 'https://dashboard.baseaim.co'`
         - `NEXT_PUBLIC_SENTRY_DSN: ${{ secrets.NEXT_PUBLIC_SENTRY_DSN }}`
       - Prepare standalone artifact:
         ```
         cp -r public .next/standalone/
         cp -r .next/static .next/standalone/.next/
         ```
       - Copy `ecosystem.config.js` into standalone:
         ```
         cp ecosystem.config.js .next/standalone/
         ```

    3. **Deploy job** (needs: build):
       - Transfer artifact to VPS using `appleboy/scp-action@v0.1.7`:
         - `host: ${{ secrets.VPS_HOST }}`
         - `username: ${{ secrets.VPS_USER }}`
         - `key: ${{ secrets.VPS_SSH_KEY }}`
         - `source: '.next/standalone/*'`
         - `target: '/var/www/dashboard'`
         - `strip_components: 2` (removes `.next/standalone/` prefix so files land directly in /var/www/dashboard/)
         - `overwrite: true`
         - `rm: true` (clean target before copy)

       Wait -- `appleboy/scp-action` puts files under target preserving structure. Use `strip_components` to flatten. Actually, the simpler approach: set `source: '.next/standalone/'` and `target: '/var/www/dashboard'` with `strip_components: 2`.

       Actually, the cleanest approach is a single SSH step that does rsync. Use `appleboy/ssh-action@v1.0.3` to run rsync from the VPS pulling from the runner -- but that requires the VPS to reach the runner, which it can't.

       Better approach: Use `appleboy/scp-action@v0.1.7` to upload the artifact, then `appleboy/ssh-action@v1.0.3` to reload PM2.

       Revised deploy steps:

       a. Upload artifact with `appleboy/scp-action@v0.1.7`:
          - `host`, `username`, `key` from secrets
          - `source: '.next/standalone/'`
          - `target: '/var/www/dashboard/_deploy'`
          - `overwrite: true`

       b. SSH to VPS with `appleboy/ssh-action@v1.0.3` to swap artifact and reload:
          ```bash
          export NVM_DIR="$HOME/.nvm"
          [ -s "$NVM_DIR/nvm.sh" ] && \. "$NVM_DIR/nvm.sh"

          cd /var/www/dashboard

          # Swap deployed artifact into place
          rm -rf .next/standalone
          mkdir -p .next
          mv _deploy/.next/standalone .next/standalone

          # Copy ecosystem.config.js from deployed artifact to app root
          cp .next/standalone/ecosystem.config.js ./ecosystem.config.js

          # Clean up temp deploy dir
          rm -rf _deploy

          # Zero-downtime reload (cluster mode rolling restart)
          pm2 reload ecosystem.config.js --env production
          ```

    4. **Important secrets required** (document in YAML comments):
       - `VPS_HOST` -- VPS IP address
       - `VPS_USER` -- SSH username (e.g., deploy or root)
       - `VPS_SSH_KEY` -- Full private SSH key content
       - `SENTRY_AUTH_TOKEN` -- For source map upload
       - `SENTRY_ORG` -- Sentry organization slug
       - `SENTRY_PROJECT` -- Sentry project slug
       - `NEXT_PUBLIC_SENTRY_DSN` -- Sentry DSN for client-side

    Use a SINGLE job (not separate build/deploy jobs) to avoid artifact transfer between jobs. The SCP step runs in the same job after the build step.
  </action>
