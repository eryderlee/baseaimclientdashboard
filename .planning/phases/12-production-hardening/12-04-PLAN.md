# Plan 12-04: Zod Validation Gap Closure (API Route Handlers)

**Phase Goal:** Dashboard is secure, reliable, and production-ready
**Gap Being Closed:** Truth 5 — Input validation using Zod on ALL user-submitted data (PARTIAL → VERIFIED)

## Context

Phase 12 verification found two API route handlers accepting raw `req.json()` without Zod schemas:

1. **`app/api/user/settings/route.ts`** — PATCH handler directly destructures 6 fields (`name`, `email`, `companyName`, `phone`, `website`, `address`) from `req.json()` with no type or length checks before `prisma.user.update()`.

2. **`app/api/messages/route.ts`** — POST handler has a manual `if (!content || !content.trim())` check for content but no schema. `receiverId` is passed directly to `prisma.message.create()` without any validation — a non-CUID value would cause a Prisma error rather than a clean 400.

Both routes already require `auth()` (authenticated users only). The fix is to insert Zod schema validation immediately after `req.json()` in each handler, replacing manual checks.

    3. Define a module-level Zod schema above the PATCH function:
       ```ts
       const updateSettingsSchema = z.object({
         name: z.string().min(1).max(100).optional(),
         email: z.string().email().optional(),
         companyName: z.string().max(200).optional().nullable(),
         phone: z.string().max(50).optional().nullable(),
         website: z.string().max(500).optional().nullable(),
         address: z.string().max(500).optional().nullable(),
       })
       ```

    4. Replace the raw destructure line:
       ```ts
       const { name, email, companyName, phone, website, address } = await req.json()
       ```
       with:
       ```ts
       const body = await req.json()
       const parsed = updateSettingsSchema.safeParse(body)
       if (!parsed.success) {
         return NextResponse.json({ error: "Invalid input" }, { status: 400 })
       }
       const { name, email, companyName, phone, website, address } = parsed.data
       ```

    5. Everything else in the function stays identical (auth check, prisma.user.update, activity log, response).

    IMPORTANT: The catch block already catches unexpected errors — do NOT add a separate try/catch for Zod. safeParse is used precisely to avoid throwing.
