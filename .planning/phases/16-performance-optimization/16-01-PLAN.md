<objective>
Configure Supabase Supavisor transaction mode connection pooling to prevent connection exhaustion in serverless (Netlify Functions) environment, and add directUrl to schema.prisma for Prisma migrations.

Purpose: Without connection pooling, each serverless function invocation creates a new PostgreSQL connection. Under concurrent load, this exhausts Supabase's connection limit causing errors. This is the highest-priority performance fix because it's also a correctness risk.

Output: schema.prisma updated with directUrl, user guided to update Netlify env vars with pooler URL.
</objective>

  <action>
1. Update the `datasource db` block in `prisma/schema.prisma` to include `directUrl`:

```prisma
datasource db {
  provider  = "postgresql"
  url       = env("DATABASE_URL")
  directUrl = env("DIRECT_URL")
}
```

The `directUrl` tells Prisma to use the direct connection for migrations/introspection while using the pooled `DATABASE_URL` for runtime queries.

2. Audit all `$transaction` usage in the codebase. Search for `$transaction` in:
   - `app/admin/actions.ts` (lines 72, 196 — interactive transactions)
   - `app/actions/auth.ts` (line 211 — batched transaction)
   - `app/admin/clients/[clientId]/actions.ts` (line 138 — batched transaction)

For each usage, verify it will work with Supavisor transaction mode:
- **Batched transactions** (`prisma.$transaction([query1, query2])`) — these work fine with pgbouncer mode because Prisma sends them as a single prepared statement batch.
- **Interactive transactions** (`prisma.$transaction(async (tx) => {...})`) — these also work with `?pgbouncer=true` because Prisma uses a single connection for the entire callback. The `pgbouncer=true` flag disables named prepared statements, which is the actual incompatibility.

Document findings as comments in the summary. No code changes needed for the transactions — they are compatible.

3. Verify the JSON is valid: `npx prisma validate`

Do NOT add DIRECT_URL to `lib/env.ts` — it's only used by Prisma CLI (migrations/push), not at runtime. The existing `DATABASE_URL` validator already accepts `postgresql://` URLs, which covers the pooler URL format.

<task type="checkpoint:human-action" gate="blocking">
  <name>Task 2: Update Netlify environment variables for connection pooling</name>
  <action>
**Step 1: Get Supavisor transaction mode connection string from Supabase**
1. Go to your Supabase project dashboard
2. Navigate to Project Settings -> Database -> Connection string
3. Select **Transaction mode** (NOT Session mode)
4. Copy the connection string — it will look like:
   `postgresql://postgres.[PROJECT_REF]:[PASSWORD]@aws-0-[REGION].pooler.supabase.com:6543/postgres`

**Step 2: Append required parameters**
Add `?pgbouncer=true&connection_limit=1` to the end of the transaction mode URL:
```
postgresql://postgres.[PROJECT_REF]:[PASSWORD]@aws-0-[REGION].pooler.supabase.com:6543/postgres?pgbouncer=true&connection_limit=1
```

CRITICAL: `?pgbouncer=true` is REQUIRED. Without it, Prisma's prepared statements will fail under load with "prepared statement already exists" errors.

**Step 3: Get direct connection string**
1. In the same Supabase page, switch to **Direct connection** (port 5432)
2. Copy this string — it will look like:
   `postgresql://postgres:[PASSWORD]@db.[REF].supabase.co:5432/postgres`

**Step 4: Update Netlify environment variables**
1. Go to Netlify -> Site configuration -> Environment variables
2. Update `DATABASE_URL` to the **transaction mode pooler URL** (from Step 2, with `?pgbouncer=true&connection_limit=1`)
3. Add a new variable `DIRECT_URL` with the **direct connection string** (from Step 3)

**Step 5: Trigger a new deploy**
1. In Netlify, trigger a redeploy (Deploys -> Trigger deploy -> Deploy site)
2. Wait for the build to complete
3. Test the live site:
   - Log in as admin — verify client list loads
   - Create a test client (tests $transaction in admin actions)
   - Log in as a client — verify dashboard loads
