1. **Optimize `getAllClientsWithMilestones()`** — replace `include` with `select` to exclude unused fields:

Current (fetches everything including `notes: Json`):
```typescript
const clients = await prisma.client.findMany({
  include: {
    user: { select: { name: true, email: true } },
    milestones: { orderBy: { order: 'asc' } },
  },
  orderBy: { createdAt: 'desc' },
})
```

Optimized:
```typescript
const clients = await prisma.client.findMany({
  relationLoadStrategy: 'join',
  select: {
    id: true,
    companyName: true,
    industry: true,
    website: true,
    phone: true,
    address: true,
    adAccountId: true,
    driveFolderId: true,
    onboardingStep: true,
    isActive: true,
    createdAt: true,
    updatedAt: true,
    userId: true,
    user: { select: { name: true, email: true } },
    milestones: {
      select: {
        id: true,
        title: true,
        description: true,
        status: true,
        progress: true,
        startDate: true,
        dueDate: true,
        completedAt: true,
        order: true,
        createdAt: true,
        updatedAt: true,
        clientId: true,
        // notes: EXCLUDED — Json field, potentially large, unused by analytics
      },
      orderBy: { order: 'asc' },
    },
  },
  orderBy: { createdAt: 'desc' },
})
```

IMPORTANT: Include all Client scalar fields in the select that the admin client list or analytics functions use. The `notes` field on Milestone is the one we're specifically excluding — it's a Json field containing an array of note objects that can grow large and is never used by `getAdminAnalytics()` or the admin client list table. Check if any consumer of `getAllClientsWithMilestones()` needs `notes`:
- `getAdminAnalytics()` — does NOT use notes (only uses status, progress, dueDate)
- Admin client list page — renders client table, does NOT display milestone notes in the list view

If `notes` IS needed by the admin milestone editing page, that page uses `getClientWithMilestones()` (a separate function), not `getAllClientsWithMilestones()`, so this optimization is safe.

2. **Add `relationLoadStrategy: 'join'`** to the following DAL functions that fetch relations:
   - `getClientWithMilestones()` — fetches client + milestones + user
   - `getClientForEdit()` — fetches client + user
   - `getClientBillingData()` — fetches client + invoices + subscriptions
   - `getAdminClientForBilling()` — fetches client + user + subscriptions

For each, add `relationLoadStrategy: 'join'` as a property in the findUnique/findMany call:

```typescript
const client = await prisma.client.findUnique({
  where: { id: clientId },
  relationLoadStrategy: 'join', // Single SQL JOIN instead of N+1
  include: { ... }
})
```

Do NOT add relationLoadStrategy to functions using `unstable_cache` (the FB functions) — those cache the raw fetch result and adding join strategy inside the cache callback won't help.
